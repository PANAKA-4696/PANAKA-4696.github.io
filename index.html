サーバーサイド技術の学習と実践のため、PHPとMySQLを使用して動的なWebカレンダーを開発しました。 ユーザーが日付をクリックすることで、その日の予定を直感的に管理（登録・編集・削除）できるWebアプリケーションです。

プロジェクトの目的: PHPを用いて自分が欲しいと思っていた、シンプルで使いやすいWebカレンダーを作成し、インターンなどの予定などに必要なURLだったりなどを格納できるようなものが欲しいと思い、作成しました。

  • 使用技術: PHP, MySQL, HTML/CSS
  • 主な機能:
    • カレンダーの月次表示（前月・次月へのナビゲーション）
    • 日付ごとの予定の登録、表示
    • 予定の編集・削除機能 (CRUD)
  • 工夫した点・アピールポイント:
    • カレンダーロジックの実装: PHPの`DateTime`クラスを利用し、うるう年や月の初日の曜日を正確に計算し、カレンダーを動的に描画するロジックを自作しました。
    • セキュリティ対策: データベース操作はすべてPDO（プリペアドステートメント）を使用し、フォームからの入力値は適切にエスケープ処理（`htmlspecialchars`など）を行うことで、SQLインジェクションやXSS（クロスサイトスクリプティング）の基本的な対策を実装しました。
    • UI/UXの工夫: カレンダーの色味が最初はシンプルなものだったのですが、そちらを見やすくするために各項目の色分けをし、また色についても視覚的にすっきりとした配色に変更しました。 各ページにおいてもレイアウトをシンプルでモダンなデザインにすることで、ユーザーが直感的に操作できるよう工夫しました。
  • 開発で苦労した点:

    カレンダーの日付マスと、データベースから取得した「その日の予定」を正しく紐づけて表示するロジックに苦労しました。最終的に、日付をキーにした連想配列に予定情報を格納することで解決しました。 最初は、その日に対する祝日の計算などに第一月曜日である日などに対するStyleなどの割り当てがむづかしく、苦労しましたが、日付を連想配列のキーにすることでその日が何月なのか、何週目なのか、何日なのか、何曜日なのかを判別し、スタイルの割り当てを行うことができました。

GoogleDrive保存・取得

Androidネイティブアプリ開発の学習と実践のため、Kotlinを使用して問診票（アンケート）アプリを開発しました。 実際に父親の使っている問診票をDX化させ、紙ベースの問診票をアプリで代替することを目的としたアプリケーションです。 それに加えてデータで管理することで、過去の問診票の履歴を簡単に参照できるようにし、顧客の管理、カルテの管理を同時に効率化することを目指しました。

課題: 私の父親が経営している接骨院では、紙ベースの問診票を使用しており、患者の情報管理や過去の問診票の参照が煩雑で非効率的でした。その紙ベースの問診票が100枚を超えたことを期に、デジタル化の必要性を感じました。そのため、父親に相談の上、問診票アプリの開発を決意しました。

プロジェクトの目的: 紙ベースの問診票をデジタル化し、患者情報の管理と過去の問診票の参照を効率化すること。また、私の家計的な問題を含めてどれだけコストのかからない方法で実現できるかを考え、アプリ開発を通じて技術力の向上を図ることを目的としました。

  • 使用技術:
    • 言語: Kotlin
    • UI: Android View System (XML)
    • データベース: SQLite (SQLiteOpenHelper)
    • API・認証: Google Drive API (OAuth 2.0), Google Auth Library
    • 非同期処理: Kotlin Coroutines
    • ライブラリ: Gson (JSONパース)
    • IDE: Android Studio
  • 主な機能:
    • 新規顧客登録機能: 新しいお客様の個人情報を登録します。
    • 問診票（カルテ）追記機能: 既存のお客様の問診内容を記録します。
    • 顧客情報・問診内容の確認機能: 登録された情報を一覧で確認・検索できます。
    • Google Driveへのバックアップ・復元機能: 端末の故障や買い替えに備え、全データをGoogle Driveに安全に保存・復元できます。
  • 工夫した点・アピールポイント:

    目的としてある、コスト面の課題を考え、デバイス自体をAndroidを選ぶことで、iOSに比べて端末価格を抑え、アプリ自体も自身で作成することで、ランニングコストを抑えることに成功しました。 さらにバックアップとしてのGoogle Driveへのエクスポート機能を実装することで、データの安全性も確保しながら、バックアップへのコストをゼロに抑えることができました。

    • 画面遷移の実装: 複数のActivity（またはFragment）間でのデータ受け渡しを`Intent`（や`Bundle`）を使い、アプリの状態を正しく管理しながら画面を遷移させるロジックを実装しました。
    • データ永続化: 回答データを端末内に保存するため、SQLite（またはRoom）を導入し、データベースの設計からCRUD処理までを自力で実装しました。
    • API連携: Google Drive APIを利用して、回答データのバックアップ機能を実装しました。これにより、ユーザーは簡単にデータをクラウドに保存・復元できるようになりました。
    • UI/UXの工夫: 私の父親が使うということを考慮し、あまりPC操作などに慣れていない方でも直感的に操作できるよう、シンプルで分かりやすいデザインを心がけました。 また、ボタン名などをしっかり考えることで、迷わず操作できるように工夫しました。
  • 開発で苦労した点:

    GoogleDriveAPIの認証フローと非同期処理の実装に苦労しました。特に、OAuth 2.0を用いたユーザー認証とトークン管理の部分で多くの試行錯誤がありました。 AIや、ネットの情報を駆使して、mainActivityでの認証の処理や、database_helperでのデータベースの作成、管理の部分をできる部分から作成していき、最終的にはbuilderなどを使用して、GoogleDriveAPIの認証フローを実装することができました。
